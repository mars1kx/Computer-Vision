## 1. Problem Description
**Автор: Lead CV Engineer (имя: __________)**

Задача — подсчитать количество уникальных людей на видео с **верхней
(статичной) камеры** и оценить **преобладающее направление их движения**.
Это тема №14 из брифа (*People Counting & Flow Direction*, домен Human Systems).

**Вход:** видеоролик с камеры наблюдения над открытой площадкой
(1280×1242, 30 fps, ~8.6 с). Нарезается в 65 кадров (каждый 4-й кадр,
эффективные ~7.5 fps).

**Выход (автоматический, интерпретируемый):**
- `total_people` — число уникальных людей за ролик;
- `dominant_direction` — преобладающее направление потока (одна из 8 сторон света);
- `by_sector` — распределение людей по 8 секторам (роза направлений).

**Ограничение брифа:** разрешён **только классический OpenCV** — без
нейросетей и предобученных детекторов. Поэтому в основе — вычитание фона
(MOG2), пороги (Otsu), морфология и трекинг по центроидам.

## 3. Pipeline Design
**Автор: Lead CV Engineer**

Система реализует все **5 обязательных стадий** брифа, соединённых в один
запускаемый скрипт `run.py`:

```
image → enhance → segment → clean → detect → decide
```

| # | Стадия | Что делает | Файл |
|---|---|---|---|
| 1 | **Enhance** | CLAHE по L-каналу LAB + bilateral denoise | `src/enhance.py` |
| 2 | **Segment** | MOG2 background subtraction + Otsu по разности с фоном | `src/segment.py` |
| 3 | **Clean** | morph open/close, фильтры по площади/форме/solidity | `src/clean.py` |
| 4 | **Detect** | связные компоненты + watershed-сплит групп + CentroidTracker | `src/detect.py`, `src/tracker.py` |
| 5 | **Decide** | подсчёт уникальных людей + 8-секторная роза направлений | `src/decide.py` |

**Поток данных в `run.py`:**
1. Загрузка 65 кадров, ресайз до 1280 по большей стороне.
2. Прогрев MOG2 (построение модели фона по всем кадрам).
3. Построение статической маски фона (постоянные «не-людские» пиксели).
4. Основной прогон: `enhance → segment → clean → temporal_smooth → detect → tracker.update`.
5. После всех кадров: `decide` → `stats.json` + `direction_rose.png`.

## 4. Methods Used (по стадиям)
**Авторы: владельцы соответствующих стадий**

### 4.1 Enhance — *Image Processing Specialist*
- **CLAHE** (Contrast Limited Adaptive Histogram Equalization) по L-каналу
  цветового пространства **LAB** — повышает локальный контраст, не искажая цвет.
- **Bilateral filter** — сглаживает шум, сохраняя края силуэтов.

### 4.2 Segment — *Image Processing Specialist*
- **MOG2** (Zivkovic, 2004) — вероятностная модель фона; камера статична,
  люди движутся → идеальный сценарий для вычитания фона.
- **Otsu** по разности `|кадр − модель фона|` — резервный канал, ловит людей,
  которые остановились (MOG2 их «впитывает» в фон).
- Otsu ограничивается зоной вокруг MOG2-детекций (`bitwise_and` с расширенной
  MOG2-маской) — убирает ложные срабатывания от теней и дальних объектов.
- Финальная маска = `MOG2 OR (Otsu ∩ dilate(MOG2))`.

### 4.3 Clean — *Morphology & Report Lead*
- **MORPH_OPEN** (ядро 3) — снимает соль-перец шум.
- **MORPH_CLOSE** (ядро 15) — склеивает разорванные силуэты.
- **connectedComponentsWithStats** + геометрические фильтры:
  - площадь `400 … 30000` px²;
  - высота ≥ 20 px;
  - circularity ≤ 0.72 (отсекает декоративные круги на полу);
  - aspect ratio `0.9 … 7.0`;
  - solidity ≥ 0.35 (отсекает «рваные» тени).
- Вычитание статической маски фона.

### 4.4 Detect + Tracking — *Lead CV Engineer + Decision & Tracking Engineer*
- **connectedComponentsWithStats** → bbox каждого пятна.
- **Watershed-сплит** «слипшихся» людей через distance transform
  (порог пика 0.55, разделение групп > 22000 px²).
- **CentroidTracker**: жадное сопоставление по расстоянию **с предсказанием
  позиции по скорости** (last + velocity), `MAX_DISTANCE=140`, `MAX_MISSED=4`.
  Поле `hits` (число реальных сопоставлений) отличает человека от фантома.

### 4.5 Decide — *Decision & Tracking Engineer*
- Человек засчитывается, если `hits ≥ 10` (реально виден ≥10 кадров).
- Направление = `atan2` между первой и последней точкой трека; если смещение
  < 80 px → `stationary` (стоит, в розу не идёт).
- Преобладающее направление = самый населённый сектор.

## 6. Failure Cases
**Автор: Data & Testing Engineer**

Честный разбор слабых мест (требование брифа):

1. **Фантомные треки (главная проблема, решена частично).**
   На ~7.5 fps трекер раньше терял ID и пересоздавал людей → завышение до 28.
   Исправлено предсказанием по скорости + подсчётом по `hits` → 15. Остаточно:
   пара коротких треков (`hits≈10`) могут быть на грани.

2. **Долго стоящие люди.**
   Если человек стоит десятки секунд, MOG2 впитывает его в фон, а Otsu
   обрезается зоной MOG2 → риск выпадения из подсчёта. На коротком ролике
   не проявляется.

3. **Ложные рамки в кадре.**
   Свежие треки (`hits=1–2`) иногда рисовались на пустой плитке. Исправлено
   фильтром отрисовки `MIN_HITS_TO_DRAW=3` (см. `src/viz.py`).

4. **Слипшиеся группы.**
   Watershed разделяет 2–3 человек, но плотные группы (4+) могут считаться
   как один или дробиться неточно.

5. **Тени и блики.**
   MOG2 помечает тени (значение 127), мы их отбрасываем, но сильные блики на
   светлом полу изредка проходят фильтры Clean.
